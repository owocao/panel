# 项目暂停记录

## 暂停时间

2026-07-17

## 为什么暂停

经过一段高强度实际使用，当前项目已经做成一套可部署、可管理的个人导航页和手工收藏夹管理器，但使用体验仍不够顺手，也没有解决最初想解决的核心痛点。

- 开发重心逐渐投入到 UI、拖拽、弹窗、设置草稿和管理细节。
- 当前产品更像导航页加手工收藏夹管理器，而不是浏览器收藏工作流的替代品。
- 继续修复小问题或反复微调视觉的实际收益已经较低。
- 现有功能、部署方式和代码结构仍有价值，因此项目只是临时暂停，未来仍可能继续。

## 当前可用状态

- `v0.3.3` 可通过 Docker 部署。
- 登录、导航、收藏夹、设置、导入导出和备份恢复等主要功能已有可用实现。
- `v0.3.3` 合并前通过前端构建、后端 `go test ./...` 和 `go vet ./...`。
- GitHub 已发布 `v0.3.3` Release。
- Docker 使用 `./data:/app/data` 持久化 SQLite、上传文件和其他运行数据。

## 最初想解决的问题

根据项目早期文档和提交历史，项目原本希望做一个单管理员使用的轻量个人导航面板与书签管理器：集中放置常用服务入口、支持内外网地址、整理浏览器书签，并通过本地 SQLite 和 Docker 保持可控部署。

## 最终做成了什么

当前是一个自托管的个人导航页和分层收藏夹管理器。它有登录、导航卡片、内外网优先选择、图片和文字图标、收藏夹抽屉、文件夹树、搜索、批量操作、拖拽、HTML 导入导出、favicon/metadata、设置、S3 图片同步、备份恢复和 Docker 部署。

它适合愿意手工维护个人入口和收藏数据的单管理员用户，不是浏览器收藏夹同步工具。

## 没解决的核心问题

- 无法自动接管或实时同步浏览器收藏夹。
- 没有浏览器扩展、自动采集或双向同步。
- 用户仍要维护第二份收藏数据，浏览器与面板的数据会自然分离。
- 导航和收藏夹没有形成统一的高频工作流。
- 手机端和跨设备体验只达到基础可用。
- 实际使用频率与维护成本未必匹配。

## 以后继续开发前先回答的问题

1. 真正要解决的核心需求是什么？
2. 数据主源应该是浏览器还是服务器？
3. 是否需要浏览器扩展？
4. 导航和收藏夹是否应该合并为同一工作流？
5. 是继续改当前项目，还是只复用后端和数据模型重做前端？
6. 哪些现有功能真实高频使用？
7. 哪些功能可以删除，而不是继续维护？

## 重启项目时的第一步

- 先部署 `v0.3.3` 并实际使用几天。
- 阅读 `docs/project-state.md` 和本文件。
- 查看 `v0.3.1` 至 `v0.3.3` 的 Release notes 和后续 Git 历史。
- 不要立即重构。
- 先重新确定一个核心痛点和最小目标。
- 创建新分支后再开始开发。

## 不建议立即做的事

- 不要只为了减少行数继续拆 `App.vue`。
- 不要重构 `useDragSort.js`。
- 不要继续反复微调 UI。
- 不要在没有产品目标前增加新功能。
- 不要移动现有稳定 tag。
