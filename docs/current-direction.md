## 1. 接手基线

本项目已经不是早期原型。后续开发以当前代码和本文档为基线，禁止因为旧文档中的早期描述回退已确认的产品方向。

当前稳定方向：

- 产品定位：单管理员个人使用的轻量导航面板与书签管理器。
- 技术路线：Go 后端、Vue 3 + Vite 前端、SQLite 本地数据库、Docker 交付。
- 首页是核心体验，优先保证加载速度、信息密度、布局稳定和操作一致性。
- 收藏夹通过左侧抽屉打开，首页首次加载不得加载完整收藏夹数据。
- 系统设置采用弹窗式设置中心，设置项优先通过草稿保存。
- 导航分组和卡片管理已经纳入系统设置草稿机制，保存后再批量写入后端。
- 搜索引擎管理、备份恢复、个性化设置都在系统设置内，不再恢复为分散入口。

后续开发时，应先读：

1. `AGENTS.md`
2. `docs/current-direction.md`
3. `docs/project-state.md`
4. `docs/requirements.md`
5. `docs/tasks.md`
6. `docs/development.md`

如文档冲突，以本文档和 `docs/project-state.md` 为当前事实基础；旧需求只作为范围参考。

## 2. 当前阶段目标

当前阶段目标不是继续堆功能，而是把项目整理成可长期维护的 V1 状态。

优先级：

1. 文档与代码事实保持一致。
2. 修复高风险保存、恢复、导入导出、鉴权和数据一致性问题。
3. 小步拆分前后端过大的文件，保持行为不变。
4. 为关键逻辑补测试和手动验收清单。
5. 最后再继续扩展新功能。

当前不优先：

- 大规模重构。
- 替换技术栈。
- 引入状态管理库。
- 重做视觉风格。
- 新增多用户、权限体系、验证码、复杂监控等非 V1 能力。

### v0.3-ui 当前方向

当前暂停继续大范围 UI 改动。下一步继续做 PC 端人工回归，确认首页、设置页、编辑弹窗、移动弹窗、右键菜单和收藏夹抽屉在深色玻璃主题下的可读性和操作稳定性。

回归后再选择单点优化：

- 设置页视觉精修。
- 首页卡片 hover / 编辑态细节。
- 统一确认弹窗，逐步替代原生 `confirm()`。
- v0.3-ui 收口和发布。

当前约束：

- 手机端不是当前主线，只要求基础可用。
- 不再继续大范围全站主题改动。
- 不再继续处理收藏夹抽屉书签拖拽蓝框问题；当前方案为仅保留插入线提示。
- 若 PC 端人工回归没有新问题，准备 v0.3-ui 收口。
- 后续每轮尽量小范围修改并及时提交。
- 不要移动 `v0.2-stable` 标签。

## 3. 已确认的产品方向

### 首页导航

- 首页保持轻量分组布局。
- 真实导航数据为空时应显示空状态，不显示测试分组或测试卡片。
- 分组默认无明显背景框和底色，编辑状态才使用边框、阴影或上浮表达。
- 卡片尺寸、间距、圆角和 hover 风格必须统一。
- 链接默认当前标签页打开。
- 卡片编辑入口优先使用右键菜单、编辑态点击等已确认方式。
- 编辑/排序状态尽量通过视觉状态表达，不新增明显模式说明文字。
- 拖拽排序采用左键直接拖动，不恢复长按作为桌面端默认排序方式。
- 卡片标题最多 15 个字，分组名称最多 10 个字。
- 卡片公网地址是必填项。
- `urlMode` 只允许 `lan`、`wan`，默认 `wan`，不恢复 `auto`。

### 内外网地址

- 当前保留手动“内网优先 / 公网优先”切换。
- 点击卡片时按优先级探测地址，超时后回退另一个地址。
- `lanDetectTimeout` 可继续作为优先地址探测超时配置。
- 不恢复 `lanDetectUrl` 和 `autoDetectLan` 自动检测入口。
- 不增加单卡片独立检测地址，除非用户重新确认进入范围。

### 系统设置

- 设置中心是弹窗式界面。
- 左侧菜单可收起，右侧内容固定区域滚动。
- 每次从首页入口打开系统设置时，默认进入首个菜单“个性化”，不继承上次停留菜单。
- 保存成功后关闭设置页。
- 设置页修改优先写入 `settingsDraft`，未保存不影响首页实际设置。
- 导航分组和卡片修改优先写入 `navGroupsDraft`，未保存不影响首页和后端数据。
- 收藏夹管理菜单中的文件夹新增、编辑、移动、排序、删除优先写入 `foldersDraft`，未保存不影响抽屉和后端数据。
- 新增设置项必须优先沿用草稿机制。

### 搜索引擎

- 搜索引擎管理是系统设置中的独立菜单。
- 首页搜索框只做外部搜索引擎跳转，不搜索本地收藏夹或导航卡片。
- 首页搜索引擎选择器展示图标，不恢复到个性化设置里。

### 备份恢复

- 导入导出能力统一放在“备份恢复”菜单。
- 不恢复独立“导入导出”菜单。
- 不恢复“备份到 S3”按钮。
- S3 保留为图片上传同步和连接测试能力。
- 导航页可单独备份和恢复。
- 全局恢复会覆盖数据，必须有二次确认。

### Logo

Logo 配置已取消。

禁止恢复：

- 显示 Logo
- 上传 Logo
- Logo 图片配置
- `showLogo`
- `logoUrl`

## 4. UI 和交互禁区

后续 UI 开发必须保持一致性，禁止出现“某个页面自己长一套”的情况。

不得回退：

- 不要恢复明显卡片容器式首页分组背景。
- 不要恢复独立导入导出菜单。
- 不要把搜索引擎管理放回个性化。
- 不要把设置项改成未保存立即影响首页。
- 不要让导航管理排序未保存即写后端。
- 不要把卡片标题限制改回 10 个字。
- 不要把卡片编辑页按钮恢复为“取消 / 保存”旧布局，除非重新设计整套弹窗交互。
- 不要在编辑模式显示“排序模式”等多余文字。
- 不要直接暴露浏览器默认 `select`、`file input` 样式。
- 不要使用原生 `prompt()` 作为表单输入。
- 尽量替换已有原生 `alert()` / `confirm()` 为统一弹窗或确认组件。

## 5. 当前代码结构方向

前端：

- `frontend/src/App.vue` 仍是主状态和业务编排中心。
- `frontend/src/components/` 已承载部分展示型组件和设置区块。
- 后续拆分必须小步进行，不引入状态管理库，不改变交互。
- 优先拆分边界清晰区域：编辑弹窗、收藏夹抽屉、导航分组区块。

后端：

- `backend/internal/httpx/server.go` 仍包含路由注册、通用工具、备份恢复、S3、上传、metadata、书签导入导出等逻辑。
- 已拆分：认证、导航、收藏夹基础接口、设置、健康检查。
- 后续拆分只做“移动代码、不改行为”的保守拆分。
- 建议顺序：metadata、资源上传、书签导入导出、导航备份恢复、全局备份恢复、S3。

数据：

- SQLite 是 V1 固定选择。
- 目前迁移方式是 `CREATE TABLE IF NOT EXISTS` 加少量兼容 SQL，没有版本化 migration。
- 修改数据库结构前必须单独说明影响范围、兼容策略和回滚方式。

## 6. 当前优先风险

后续开发优先处理这些问题：

1. 系统设置保存和导航草稿保存不是同一个后端事务，存在部分成功风险。
2. 系统设置内收藏夹草稿保存仍由前端拆分为多个 HTTP 请求；删除收藏夹保存用户实测仍约 6-7 秒，未达到 1 秒级目标。
3. 收藏夹接口的后端校验和不存在资源的 404 处理弱于导航接口。
4. 全局备份恢复缺少版本信息和更细的恢复策略。
5. 前端仍有原生 `confirm()` 和 `alert()`，应逐步替换为统一弹窗。
6. 文档曾多处过期，后续大改后必须同步文档。
7. Go 和 Node/npm 使用项目内隔离工具链，执行验证命令时需要把 `.tools/go/bin` 和 `.tools/node/bin` 加入 `PATH`。

## 7. 后续工作方式

每次开发应遵循：

1. 先确认需求是否属于 V1。
2. 先读相关文档和现有实现。
3. 小步修改，避免一次性重构多个边界。
4. 修改后按 `docs/development.md` 执行验证；如果环境缺命令，要明确说明。
5. 较大功能完成后同步 `current-direction`、`project-state`、`tasks` 或 `development`。

当前项目已经交由 Codex 后续维护。后续以“稳定、可维护、文档同步”为第一目标。
