## 1. 当前阶段原则

后续开发不再按早期“原型优先”的节奏推进。项目已经具备主要功能，当前应以稳定、可维护、可验证为主。

优先级：

1. 文档同步和接手基线。
2. 高风险流程修复。
3. 后端校验和测试补齐。
4. 前端交互一致性修复。
5. 保守拆分大文件。
6. Docker 和发布流程复核。

不优先：

- 大规模 UI 重做。
- 大规模架构重构。
- 多用户系统。
- 权限分级。
- 验证码。
- 服务器监控、容器管理等非 V1 能力。

## 3. P1 高风险流程

### GitHub 拉取后 Docker 部署问题

已处理：

- 新部署或真实导航数据为空时，不再显示内置测试分组和测试卡片。
- 删除所有导航分组后，保存不会再恢复默认测试数据。
- 删除包含卡片的分组时，确认文案改为“分组内存在卡片，点击确定将删除该分组及内部所有卡片；点击取消放弃删除”。
- 系统设置“关于”文案改为“个人自用导航站和网页收藏夹。”
- 系统设置内导入或删除收藏夹后，再点击保存，不再触发无关的导航草稿保存。
- 系统设置“收藏夹”菜单增加移动、上移、下移按钮。
- 系统设置“收藏夹”菜单的文件夹新增、编辑、移动、上移、下移、删除改为草稿操作，点击保存后才写入后端，关闭设置页会丢弃草稿。
- 系统设置保存增加保存中状态和重复提交保护；收藏夹草稿保存只提交变化过的文件夹，减少无意义请求。
- 系统设置每次从首页入口打开时默认进入“个性化”，避免保存后再次打开停留在空的收藏夹草稿菜单。
- API 返回未登录时，前端会关闭设置/抽屉并回到登录页，提示“登录已失效，请重新登录”。
- 首次打开页面进入鉴权启动态，确认登录后才渲染首页，避免先显示首页再跳登录页。
- SQLite 连接池限制为单连接，降低登录、保存等并发写入触发 `SQLITE_BUSY` 的概率。
- 收藏夹草稿保存删除文件夹时只提交最上层删除项，依赖 SQLite 级联删除内部子目录和书签；文件夹写操作串行提交，避免并发写导致锁等待和前端超时。
- 编辑收藏夹弹窗增加“移动”下拉框，可移动到根目录或其他收藏夹下。
- 收藏夹抽屉内收藏夹右键菜单增加“移动”入口。
- 收藏夹抽屉、系统设置窗口和编辑/移动弹窗拦截鼠标滚轮事件，避免滚动穿透到首页。
- 打开系统设置时会关闭收藏夹/首页右键菜单，避免菜单残留。

仍需手动验证：

- GitHub 重新拉取后用 Docker Compose 全新部署，首页不出现测试数据。
- 删除全部分组并保存后，刷新页面仍为空导航状态。
- 收藏夹移动到根目录、同级目录、其他父目录均符合预期。
- 收藏夹不能移动到自身或子收藏夹下。

仍未达标：

- 系统设置“收藏夹”中删除收藏夹后点击保存，当前已避免超时和明显 SQLite 锁错误，但用户实测仍需约 6-7 秒，目标应压到 1 秒左右。
- 下一步需要继续定位后端实际耗时，优先考虑新增事务型收藏夹草稿保存接口或专用批量删除/排序接口，避免前端用多个 HTTP 请求拼装保存流程。

### 设置保存与导航草稿保存

问题：

- 前端 `submitSettings()` 先保存系统设置，再批量保存导航草稿。
- 导航草稿保存由多个 API 请求组成。
- 中途失败时可能出现设置已保存、导航只保存一部分的状态。

建议：

- 后端新增事务型接口，接收完整设置草稿和导航草稿，或至少新增事务型导航批量保存接口。
- 前端保存设置时先验证草稿，再一次性提交。
- 失败时不关闭设置页，并给出明确错误。

验收：

- 新增、编辑、删除、排序分组和卡片后，一次保存成功。
- 任一校验失败时，后端不产生部分导航修改。
- 前端显示失败原因，设置页保持打开。

### 全局备份恢复

问题：

- 全局恢复会覆盖 `data/` 下文件。
- 缺少版本信息校验和恢复内容摘要。

建议：

- 备份包包含版本信息和 manifest。
- 恢复前后端验证 manifest。
- 前端恢复前展示将覆盖的内容。
- 保留二次确认。

验收：

- 备份包可识别版本。
- 版本不一致时提示风险但允许确认继续。
- 恢复不会写出 `data/` 目录之外。

### 书签导入导出

问题：

- 需要持续用真实 Chrome/Safari 样例验证。
- 需要覆盖重复判断规则。

建议：

- 增加导入解析测试样例。
- 增加导出格式测试。
- 明确首次导入不内部去重，后续导入按 URL、标题、备注、所属目录判断重复。

验收：

- Chrome 导出文件可导入并保持结构。
- Safari 导出文件可导入并保持结构。
- 导出的 HTML 可被浏览器识别。

## 9. v0.3-ui 页面体验优化计划

> 基线：`v0.2-stable` 之后的 `master`
> 范围：只做前端 UI/交互体验优化，不做后端功能、不做性能优化、不改变数据结构。
> 原则：小步推进、每轮可人工回归；优先保持现有草稿机制、懒加载机制和模块边界。

### v0.3-ui 设计约束补充

- 操作按钮优先保持外露，不默认收纳到“更多”菜单。
- 收藏夹管理、分组管理等高频管理页面，保留直接操作按钮，减少二次点击。
- 不追求极简到隐藏操作入口，优先保证非开发用户可直接理解和操作。
- 页面整体需要适配深色背景图片场景。
- 所有面板、卡片、按钮、弹窗、输入框、菜单都要考虑背景图上的可读性。
- 避免大面积纯白、强灰后台风格，优先使用半透明、毛玻璃、柔和描边、适度阴影。
- 主题色应贯穿按钮、选中态、hover、焦点态、开关、标签等状态。
- 深色背景下，文字、边框、按钮需要保持足够对比度。
- UI 优化不得牺牲现有操作效率。
- 不要把收藏夹管理、分组管理的操作按钮收纳到更多菜单。
